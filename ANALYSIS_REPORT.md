# Анализ системы мониторинга доставляемости

## Что проверено в контексте HTML

По выгрузке чата восстановлена архитектура:
- контрольные Gmail seed-ящики;
- сбор статусов по кампаниям;
- Cloud Run/Gmail API вариант и IMAP fallback;
- целевые статусы: `INBOX`, `SPAM`, `NOT_DELIVERED` + продвинутый `DELIVERED_NOT_INBOX`.

## Найденные проблемы (по логике предыдущей реализации)

1. Поиск только по `subject` создаёт ложные совпадения при 30+ кампаниях.
2. Непрозрачный приоритет статусов, если совпадения есть и в `INBOX`, и в `SPAM`.
3. Ошибка одного seed могла портить общий запуск.
4. Узкая проверка папки спама (локализация Gmail/Google Mail).
5. Нет полноценного аудита запусков и ошибок на уровне каждой проверки.
6. Не хватало пользовательского UI для ежедневной операционной работы.

## Что исправлено в этой версии

- Добавлен `cid_token` как приоритетный идентификатор кампании.
- Введено окно проверки (`window_hours`) и поддержка `date_from/date_to`.
- Выбор статуса по более свежему письму, если матч в нескольких папках.
- Добавлены алиасы spam/all-mail папок.
- Ошибки авторизации изолируются как `ERROR_AUTH` на конкретной связке campaign+seed.
- Добавлены таблицы `runs` и `run_results` для наблюдаемости.
- Реализован web dashboard с KPI и журналом запусков.

## Критерий “система готова”

Система считается готовой, когда:
- в таблице `seeds` заполнены рабочие app-password и `is_active=1`;
- кампании заполнены с `cid_token` и `subject`;
- `POST /api/run` возвращает `ok=true` и пишет строки в `run_results`.
